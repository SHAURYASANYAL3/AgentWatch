### Overview
With the introduction of Premium features (AgentWatch Cloud / Enterprise), it is critical to ensure that unauthorized users cannot simply patch the CLI to bypass entitlement checks. We need robust anti-tampering and license enforcement mechanisms to protect the monetization model from being bypassed.

### Threat Model
- Malicious users modifying the local Python source code of the CLI to strip out `if not is_premium():` checks.
- Users sharing valid license tokens or intercepting local entitlement responses.
- Reverse-engineering of client-side validation logic.

### Objectives
1. **Cryptographic Token Verification**: Premium entitlement tokens must be cryptographically signed (e.g., JWT with RS256) by the backend and verified by the CLI using a hardcoded public key.
2. **Server-Side Enforcement (Recommended)**: The most secure way to prevent patching is to ensure that premium features *require* backend processing. Instead of a local Boolean check, the CLI should send the payload to the backend, which validates the token and returns the premium result. If the backend is bypassed, the feature simply cannot function locally.
3. **Anti-Tamper & Obfuscation**: Where local enforcement is strictly necessary (due to latency or offline requirements), apply obfuscation and runtime integrity checks to the core validation modules.
4. **Device/Hardware Binding**: Bind the generated license token to a specific hardware ID or machine footprint to prevent token sharing across multiple users.

### Acceptance Criteria
- [ ] Premium license tokens are cryptographically verified.
- [ ] Bypassing local checks does not unlock premium features (enforced server-side where possible).
- [ ] Attempting to use a single license concurrently across unlinked devices flags the account for abuse.
- [ ] The CLI codebase structure prevents trivial Boolean bypasses for critical paid functionality.
